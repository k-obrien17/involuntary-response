Rewrite the GET /:slug route handler (currently lines 368-481) to use batchLoadPostData instead of 7 serial queries. The current flow is:

2. Query embed (serial)
3. Query tags (serial)
4. Query artists (serial)
5. Query like count (serial)
6. Query user liked (serial)

Replace steps 2-6 with a single call: `batchLoadPostData([post.id], req.user?.id)`. This returns embedMap, tagMap, artistMap, likeCountMap, likedByUserMap, commentCountMap -- all keyed by post.id.

Keep the comments query (step 7) as a separate query because the single-post endpoint returns full comment objects with author info and canDelete, which batchLoadPostData does not provide (it only returns counts).

- `likedByUser: !!likedByUserMap[post.id]`

The response must still include: id, slug, body, authorId, status, createdAt, updatedAt, publishedAt, author (object), embed, tags, artists, likeCount, likedByUser, comments (array with canDelete).

This reduces queries from ~8 to 3 (post, batch, comments).

**Server (server/routes/profile.js):**

In the GET /:username/profile handler, add cursor pagination matching the pattern used in GET /posts (the feed endpoint):

1. Import `parseCursor` from `../lib/post-helpers.js` (already imports batchLoadPostData, formatPosts, emailHash -- just add parseCursor to the destructure).
2. Parse `limit` from query: `const limit = Math.min(Math.max(parseInt(req.query.limit) || 20, 1), 50);`
3. Parse cursor: `const { cursorClause, cursorParams } = parseCursor(req.query.cursor);`
4. Update the posts query to include cursor clause and LIMIT:
   ```sql
   SELECT p.id, p.slug, p.body, p.created_at, p.updated_at, p.published_at,
          u.display_name AS author_display_name, u.username AS author_username, u.email AS author_email
   FROM posts p
   JOIN users u ON p.author_id = u.id
   WHERE p.author_id = ? AND p.status = 'published' ${cursorClause}
   ORDER BY p.published_at DESC, p.id DESC
   LIMIT ?
   ```
   Pass params: `user.id, ...cursorParams, limit + 1`
5. Check hasMore: `const hasMore = rows.length > limit; if (hasMore) rows.pop();`
6. Build nextCursor: `const lastPost = rows[rows.length - 1]; const nextCursor = hasMore && lastPost ? \`${lastPost.published_at}|${lastPost.id}\` : null;`
7. Response changes from `res.json({ user: {...}, posts })` to `res.json({ user: {...}, posts, nextCursor })`.

1. Add state: `const [nextCursor, setNextCursor] = useState(null);` and `const [loadingMore, setLoadingMore] = useState(false);`
2. In the fetchProfile effect, also capture nextCursor: `setNextCursor(res.data.nextCursor);`
3. Add loadMore function (same pattern as Home.jsx):
   ```javascript
   const loadMore = async () => {
     if (!nextCursor || loadingMore) return;
     setLoadingMore(true);
     try {
       const res = await profile.get(username, { cursor: nextCursor });
       setPosts((prev) => [...prev, ...res.data.posts]);
       setNextCursor(res.data.nextCursor);
     } catch (err) {
       console.error('Failed to load more:', err);
     } finally {
       setLoadingMore(false);
     }
   };
   ```

   ```jsx
         disabled={loadingMore}
         className="text-gray-500 dark:text-gray-400 hover:text-gray-900 dark:hover:text-gray-100 text-sm transition"
       >
         {loadingMore ? 'Loading...' : 'Older posts'}
       </button>
     </div>
   )}
   ```
5. In the useEffect reset, also reset: `setNextCursor(null);`
