**server/db/index.js** -- Append migration 5 to the migrations array (after migration 4, id: 5, name: 'add_post_status_likes_comments'). The SQL must be a single multi-statement string executed via db.exec() (same pattern as migrations 1-4). Contents:

```sql
ALTER TABLE posts ADD COLUMN status TEXT NOT NULL DEFAULT 'published';
ALTER TABLE posts ADD COLUMN published_at DATETIME;
UPDATE posts SET published_at = created_at WHERE published_at IS NULL;
CREATE TABLE IF NOT EXISTS post_likes (
  post_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (post_id, user_id),
  FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
CREATE TABLE IF NOT EXISTS post_comments (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  post_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  body TEXT NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_posts_status ON posts(status);
CREATE INDEX IF NOT EXISTS idx_posts_published_at ON posts(published_at DESC);
CREATE INDEX IF NOT EXISTS idx_post_likes_post ON post_likes(post_id);
CREATE INDEX IF NOT EXISTS idx_post_comments_post ON post_comments(post_id, created_at);
```

Key details:
- `status TEXT NOT NULL DEFAULT 'published'` -- existing rows auto-get 'published' at read time (SQLite stores default in schema, no row rewrite). Do NOT run a separate UPDATE for status.
- `published_at DATETIME` -- nullable (drafts will have NULL). The UPDATE backfills existing rows with their created_at value so cursor pagination works immediately.
- `post_likes` uses composite PK (post_id, user_id) -- enforces one like per user per post at DB level.
- `post_comments` is flat (no parent_id) -- threading is out of scope per REQUIREMENTS.md.
- Both tables have ON DELETE CASCADE for posts and users.

**server/middleware/auth.js** -- Add a new named export `requireContributor` after the existing `generateToken` function:

```javascript
export function requireContributor(req, res, next) {
  if (!req.user || (req.user.role !== 'contributor' && req.user.role !== 'admin')) {
    return res.status(403).json({ error: 'Contributor access required' });
  }
  next();
}
```

This is used after `authenticateToken` in the middleware chain. It checks the JWT-decoded user's role. It is not wired to any route in this plan -- that happens in Phase 11.

The file must export these 4 functions:

1. **emailHash(email)** -- MD5 hash of trimmed, lowercased email. Same implementation as currently in browse.js line 5.

2. **batchLoadPostData(postIds)** -- Identical to browse.js lines 13-59. Takes an array of post IDs, returns `{ embedMap, tagMap, artistMap }`. Imports `db` from `'../db/index.js'`. No changes to the query logic.

3. **formatPosts(rows, embedMap, tagMap, artistMap)** -- Based on browse.js lines 64-79, but with TWO additions to the response shape:
   - Add `updatedAt: p.updated_at` (currently present in posts.js response but missing from browse/search)
   - Add `publishedAt: p.published_at` (new field from migration 5)
   - Keep all existing fields: id, slug, body, createdAt (from p.created_at), author (displayName, username, emailHash), embed, tags, artists

4. **parseCursor(cursor)** -- Based on browse.js lines 85-92, but with the cursor column CHANGED from `created_at` to `published_at`:

   - New: `'AND (p.published_at < ? OR (p.published_at = ? AND p.id < ?))'`
   - This is the critical change that makes feed ordering work correctly when posts are published after being drafted.

The file must use `import { createHash } from 'crypto'` and `import db from '../db/index.js'` at the top. Use named exports (not default export).
