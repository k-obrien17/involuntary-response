  <action>
Update `client/vercel.json` to add an `/api/:path*` rewrite that proxies to the Render backend.

The Render backend URL should be read from the `RENDER_BACKEND_URL` environment variable (set in Vercel project settings). Use Vercel's environment variable syntax in the destination: `"destination": "https://$RENDER_BACKEND_URL/api/:path*"`.

IMPORTANT: Vercel does not support env vars in rewrite destinations via `$VAR` syntax. Instead, the destination must be a hardcoded URL OR the env var must be set via Vercel Edge Config. The correct approach for Vercel rewrites is to hardcode the Render URL directly. However, since we do not want to commit secrets, use a placeholder that the user will replace.

Actually — Vercel DOES support environment variables in `vercel.json` rewrites using the syntax `$ENV_VAR_NAME` in the destination field, but ONLY for Pro/Enterprise plans. For Hobby plans, the URL must be hardcoded.

The cleanest approach: hardcode the Render backend URL in `vercel.json`. The URL is not a secret (it is a public API endpoint). The production Render URL follows the pattern `https://APP-NAME.onrender.com`.

Since we do not know the exact Render URL, add a placeholder `https://RENDER_BACKEND_URL` that the user must replace with their actual Render service URL (e.g., `https://involuntary-response-api.onrender.com`).

Final `vercel.json`:
```json
{
  "rewrites": [
    { "source": "/posts/:slug", "destination": "/api/og" },
    { "source": "/api/:path*", "destination": "https://RENDER_BACKEND_URL/api/:path*" },
    { "source": "/(.*)", "destination": "/" }
  ]
}
```

Order matters:
1. `/posts/:slug` → OG tag handler (existing, must stay first)
2. `/api/:path*` → Render backend proxy (NEW)
3. `/(.*)`  → SPA catch-all (existing, must stay last)
