<objective>
Add industry-standard security headers to all Express responses, reduce JWT token lifetime from 365 days to 30 days, and add lightweight origin validation as defense-in-depth CSRF protection.

Purpose: Harden the production API server before launch. Security headers prevent clickjacking, MIME sniffing, and XSS. Shorter JWT expiry limits the window of token compromise. Origin validation provides belt-and-suspenders protection on state-changing endpoints, even though the existing JWT-in-Authorization-header pattern is already CSRF-resistant (browsers don't auto-send Authorization headers cross-origin).

Output: Security middleware module, updated Express app configuration, updated JWT expiry.
</objective>

<task type="auto">
  <name>Task 1: Install helmet and add security headers middleware with CSP</name>
  <files>server/package.json, server/middleware/security.js, server/index.js</files>
  <action>
1. Install helmet: `cd server && npm install helmet`

2. Create `server/middleware/security.js` with two exports:

   **securityHeaders middleware** — Uses helmet with these overrides:
   - `contentSecurityPolicy.directives`:
     - `defaultSrc`: `["'self'"]`
     - `frameSrc`: `["'self'", "https://open.spotify.com", "https://www.youtube.com", "https://player.vimeo.com", "https://w.soundcloud.com", "https://*.bandcamp.com", "https://embed.music.apple.com"]`
     - `scriptSrc`: `["'self'"]` (no inline scripts needed — this is an API server)
     - `styleSrc`: `["'self'", "'unsafe-inline'"]` (allow inline styles from embed HTML)
     - `imgSrc`: `["'self'", "data:", "https:"]` (allow embed thumbnails from any HTTPS source)
     - `connectSrc`: `["'self'"]`
   - `frameguard`: `{ action: 'deny' }` (sets X-Frame-Options: DENY — API responses should never be framed)
   - `xContentTypeOptions`: true (X-Content-Type-Options: nosniff — default in helmet)
   - `xXssProtection`: true (sets X-XSS-Protection: 0 — modern helmet uses 0 because the XSS auditor causes more problems than it solves. This is correct per OWASP. If the roadmap strictly requires "1; mode=block", override with `helmet.xXssProtection({ setOnOldIE: true })` but note: helmet v8+ removed this. Instead set the header manually: `res.setHeader('X-XSS-Protection', '1; mode=block')`)

   IMPORTANT: Check which version of helmet is installed after `npm install`. If helmet 8+, `xXssProtection` is no longer included. In that case, add a manual `res.setHeader('X-XSS-Protection', '1; mode=block')` in the securityHeaders middleware after the helmet() call. The success criteria says the header must be present.

   **validateOrigin middleware** — Defense-in-depth origin check for state-changing requests:
   - Only applies to POST, PUT, PATCH, DELETE methods
   - Reads `Origin` header from request
   - If no Origin header present, allow the request (server-to-server, curl, etc.)
   - If Origin IS present, check it matches `process.env.FRONTEND_URL` or `http://localhost:5173` (dev)
   - If Origin does NOT match, return 403 `{ error: 'Forbidden' }`
   - This is defense-in-depth — the real CSRF protection is that JWT is sent via Authorization header, which browsers don't auto-attach cross-origin

3. In `server/index.js`, import and register the middleware:
   - Add `import { securityHeaders, validateOrigin } from './middleware/security.js';`
   - Add `app.use(securityHeaders);` BEFORE `app.use(cors(...))` (helmet should set headers before CORS processes)
   - Add `app.use(validateOrigin);` AFTER `app.use(cors(...))` but BEFORE routes

Why helmet before CORS: Helmet sets response headers unconditionally. CORS may short-circuit on preflight OPTIONS. Order ensures all responses get security headers.

<task type="auto">
  <name>Task 2: Reduce JWT expiry from 365 days to 30 days</name>
  <files>server/middleware/auth.js</files>
  <action>
In `server/middleware/auth.js`, change the `generateToken` function:

```javascript
export function generateToken(user) {
  return jwt.sign(
    { id: user.id, email: user.email, role: user.role, username: user.username },
    JWT_SECRET,
    { expiresIn: '30d' }  // Changed from '365d'
  );
}
```

That is the ONLY change. Do NOT modify `authenticateToken` or `optionalAuth` — those use `jwt.verify()` which validates against the expiry time encoded in each individual token. Tokens already issued with 365-day expiry will continue to work until their encoded expiry date passes. New tokens get 30-day expiry. No forced logout, no migration needed.

Do NOT add refresh token logic — that is explicitly out of scope per REQUIREMENTS.md.
