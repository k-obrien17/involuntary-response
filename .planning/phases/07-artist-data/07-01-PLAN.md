  <action>
  1. Create `server/lib/apple-music.js` with a `getArtistsForAppleMusicUrl(url)` function:
     - Parse the Apple Music URL to extract region, type (album/song), collectionId, and optional songId using the same regex pattern from oembed.js: `/https?:\/\/music\.apple\.com\/([a-z]{2})\/(album|playlist|song)\/[^/]+\/([0-9]+)(\?i=([0-9]+))?/`
     - Use the iTunes Search API (public, no auth needed) to look up artist info:
       - For songs (has songId): `https://itunes.apple.com/lookup?id=${songId}&entity=song`
       - For albums: `https://itunes.apple.com/lookup?id=${collectionId}&entity=album`
     - Parse the response: iTunes API returns `results[]` array. The first result has an `artistName` field.
     - Return `[{ name: result.artistName }]` (array for consistency with Spotify function signature). No spotifyId equivalent for Apple Music.
     - Return empty array on any failure (non-fatal, same pattern as Spotify).
     - Add 5-second AbortSignal timeout on the fetch call.
     - Export `parseAppleMusicUrl(url)` as a named export too (returns `{ region, type, collectionId, songId }` or null).

  2. Add a migration to `server/db/index.js` (append to migrations array, next id after existing):
     - Add a `source` TEXT column to `post_artists` with default value 'spotify':
       `ALTER TABLE post_artists ADD COLUMN source TEXT DEFAULT 'spotify'`
     - This tracks whether artist data came from 'spotify', 'apple', or 'manual'.
     - Existing rows default to 'spotify' which is correct since only Spotify extraction existed before.

  No new npm dependencies. iTunes Search API is public REST (no API key). Use native fetch.
